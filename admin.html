Firestore Setup

Connect this admin panel and the storefront to a real Firebase backend

1. Create the Firebase project

Go to the Firebase Console → Add project → name it sk-sarees → enable Google Analytics (optional). Then add a Web App from Project Settings and copy the config object it gives you.

2. Enable the services this app needs

  • Authentication → Sign-in method → enable Phone (for OTP login on the storefront).
  • Firestore Database → Create database → start in production mode → pick a region close to India (e.g. asia-south1).
  • Storage → enable, for uploading real product photos instead of the demo SVG swatches.
  • Cloud Messaging (FCM) → for order-status and offer push notifications.

3. Collection structure

Recommended top-level collections — this mirrors the data already used by index.html and admin.html:

products/{productId}
  name, category, fabric, price, mrp, discount,
  stock, color, occasion, images[], careInstructions, createdAt

orders/{orderId}
  customerPhone, customerName, address, items[ {productId, qty, price} ],
  total, payment, status, createdAt, statusHistory[ {status, at} ]

customers/{phone}
  name, phone, addresses[], defaultAddressIndex, createdAt

categories/{categoryId}
  name, imageUrl, sortOrder

4. Suggested security rules (starting point)

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /products/{id} { allow read: if true; allow write: if request.auth.token.admin == true; }
    match /categories/{id} { allow read: if true; allow write: if request.auth.token.admin == true; }
    match /orders/{id} {
      allow create: if request.auth != null;
      allow read, update: if request.auth != null &&
        (request.auth.token.admin == true || resource.data.customerPhone == request.auth.token.phone_number);
    }
    match /customers/{phone} {
      allow read, write: if request.auth != null &&
        (request.auth.token.admin == true || request.auth.token.phone_number == phone);
    }
  }
}

Set a custom claim admin: true on your own staff accounts via a Cloud Function so only admins can change order status and edit the catalog.

5. Swap localStorage calls for Firestore calls

Right now both index.html and admin.html read/write orders with localStorage (key sk_orders) purely so this demo works without a server. To go live, replace those calls with the snippets below — the on-screen tables/forms don't need to change, only where the data comes from. 

// Load orders for admin panel (realtime)
import { collection, onSnapshot, doc, updateDoc, orderBy, query } from "firebase/firestore";

const q = query(collection(db, "orders"), orderBy("createdAt", "desc"));
onSnapshot(q, (snap) => {
  orders = snap.docs.map(d => ({ id: d.id, ...d.data() }));
  renderOrdersTable();
});

// Update order status (replaces changeStatus's localStorage write)
async function changeStatus(orderId, newStatus) {
  await updateDoc(doc(db, "orders", orderId), { status: newStatus });
}

// Save a product edit (replaces the product-overrides write)
import { doc, setDoc } from "firebase/firestore";
await setDoc(doc(db, "products", productId), productData, { merge: true });

6. Build & deploy

For a static hosting deploy: npm i -g firebase-tools → firebase login → firebase init hosting (point it at the folder containing index.html and admin.html) → firebase deploy. Keep admin.html on a separate path (e.g. /admin) protected by Firebase Auth + the admin custom claim above, so customers can't reach it.
